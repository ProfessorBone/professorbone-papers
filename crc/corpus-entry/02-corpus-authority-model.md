The CRC corpus is a multi-artifact constitution. Different files carry different kinds of public force. The primary class of an artifact tells readers and maintainers how to interpret it.

## One Primary Class

Each public artifact receives exactly one primary class. It may also declare secondary functions and an authority effect. Secondary functions do not create a second primary class.

## Classes

- Normative artifacts carry doctrine. Examples include the Core and defining companion papers.
- Governance artifacts define how the corpus operates and may carry a bounded authority effect. CRC-SSR is the worked example: it has binding canonical-location and operative-version authority, but it does not become a normative artifact.
- Evidentiary artifacts record validation, audit, or gate evidence. Receipts do not create doctrine by themselves.
- Navigational artifacts control discoverability. README files can mislead if stale, but they do not override internal authority statements.
- Explanatory artifacts explain the corpus without replacing doctrine.
- Operational templates standardize work without settling substantive questions.

## Authority Sources

The Core and defining papers carry runtime doctrine and object definitions. CRC-SSR v1.1 records authoritative location and operative version for shared load-bearing objects. Receipts record evidence. Guides govern process only within their declared scope after ratification by their identified ratifying authority. This guide is owned by Professor Bone Lab and becomes operative only through explicit ratification by Faheem Downs and publication through the governed apply process.

## Context Is Not Delegation

An existing artifact may provide relevant governance context without delegating authority. A declaration must distinguish ownership, ratification, relevant context, and actual delegated authority.
